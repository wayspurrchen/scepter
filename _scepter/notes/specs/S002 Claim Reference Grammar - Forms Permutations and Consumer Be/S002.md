# S002 - Claim Reference Grammar — Forms, Permutations, and Consumer Behavior

## Overview

This specification consolidates the canonical behavior of SCEpter's claim reference grammar across every recognized reference shape, every recognized definition shape, every metadata permutation, and every consumer that reads or produces these shapes. It is the single contract that the parser, claim-tree builder, claim-reference parser, claim index, linter, traceability matrix, gap detector, staleness checker, CLI commands, VS Code surfaces (hover, decoration, diagnostics, preview), and source code scanner all conform to.

Today the contract is distributed across multiple authoritative sources: {R004} (base grammar), {R005} (metadata, lifecycle, derivation), {R007} (search), {R008} (folder-note aggregation), {R009} (metadata key-value store), {R011} (cross-project alias prefix), and {R012} (range expansion, adjacent-section binding, hover/decoration/preview consumers). Each requirement is internally coherent. None of them cross-tabulate the full grammar against the full consumer set. Edge cases (definitions inside table cells, references inside code spans, paragraph-claims that look like cross-document references, alias-prefixed ranges with metadata, and folder-note section collisions) currently rely on consumer-by-consumer reasoning to resolve. This spec is the cross-tab.

This is a Specification — a behavioral contract — derived from those Requirements. It does not introduce new requirements except where the cross-tab itself reveals a structural gap, in which case the gap is flagged in Design Decisions for promotion to a Requirement. It does propose one extension as a labeled future addition: §8 Self-Prefixed Definition Behavior (currently unimplemented; a candidate Requirement in its own right).

### Scope

In scope:
- All reference shapes (braced and braceless) the parser currently accepts
- All definition shapes (heading, paragraph, table-row, bold-wrapped) the claim-tree builder currently recognizes
- All metadata permutations (importance, lifecycle, derives=, superseded=, freeform tags, key-value)
- Range expansion grammar and its interaction with each form
- Adjacent-section binding and its source-text gap rules
- Cross-project alias prefix grammar and its citation-not-federation invariant
- Folder-note aggregation rules for sub-file claim merging
- Per-consumer behavior: parser, tree builder, reference parser, index, linter, traceability matrix, gaps, staleness, CLI surfaces (`show`, `trace`, `gaps`, `thread`, `search`, `lint`), VS Code surfaces (hover, decoration, diagnostics, preview), source code scanner

Out of scope:
- The CLI argument grammar (`--format`, `--depth`, etc.) — that is a CLI UX concern, not a reference grammar concern
- The bidirectional reference graph storage format — internal to the reference manager
- The `@implements` / `@validates` / `@depends-on` / `@addresses` / `@see` annotation semantics in code, except as they consume reference-grammar shapes
- Verification event store format (R005's verification.json schema)
- Status validation, note type configuration, discovery paths — orthogonal subsystems
- The detailed-design-level module wiring (covered by {DD001} and successors)

Non-goals:
- This spec does not define module structure, file layout, or implementation mechanism. It defines what each consumer promises, not how the promise is realized.
- This spec does not re-litigate any settled grammar decision from upstream Requirements. Where {R004} says the dot is mandatory, this spec restates the consequence; it does not reopen the decision.
- This spec does not enumerate VS Code-specific UI affordances beyond the behavior of reference resolution and rendering. Hover layout details, color palettes, and badge styles are outside the grammar contract.

### Principle alignment

Three architectural invariants from upstream notes constrain every contract in this spec. Each is restated here in summary form; the upstream Requirement is the authority.

| Principle | Source | Restated consequence for this spec |
|---|---|---|
| Compute, don't maintain | {R004} | Every contract in this spec is over derived state. No consumer maintains an authoritative reference table separate from what is derivable from notes and source files. |
| Citation, not federation | {R011} | Cross-project alias-prefixed references are read-only display pointers. Peer claims do NOT enter the local index, derivation graph, gap report, or trace matrix. |
| Trace tracks expressions, not truth | {R004}, {R005}, claims.md | The trace matrix records that an expression of a claim exists at a projection. It does not assert the claim is satisfied. Consumer behaviors in this spec respect this distinction. |

## At a Glance

### One end-to-end worked example

Author has two notes. {R049} contains a mixture of definition forms; {R079} cites one of those definitions through a reference.

**`R049.md` (definition side, described).** A Requirement note titled "Migration Lock Authority" with two sections: `## §1 Lock semantics` and `## §3 Session-bounded locking`. Section 1 contains a level-3 heading-form claim with the bare id `AC.01` whose body asserts lock exclusivity. Section 3 contains two paragraph-form claims using the bold-wrapped form: a bare-id `LOCK.03` for session-duration locking, and a section-qualified `§3.LOCK.04` rendered as the first cell of a markdown table row asserting the abort-release behavior.

**`R079.md` (reference side, described).** A Requirement note containing prose that cites two of the R049 claims via braced fully-qualified references: a single citation of `R049.§3.LOCK.03` and a range citation `R049.§1.AC.01-02`.

**What flows through the pipeline:**

1. **`claim-tree.ts` parses `R049.md`** and produces three claim nodes:
   - `R049.1.AC.01` from the heading-form definition (§1.AC.01 line)
   - `R049.3.LOCK.03` from the bold-paragraph-form definition (`**LOCK.03**:`)
   - `R049.3.LOCK.04` from the table-row-form definition (first cell `**§3.LOCK.04**`)

1. **`claim-parser.ts` parses `R079.md`** and produces two `ClaimReference`s:
   - `{R049.§3.LOCK.03}` → `{noteId: "R049", sectionPath: [3], claimPrefix: "LOCK", claimNumber: 3}`
   - `{R049.§1.AC.01-02}` → range expansion to `R049.§1.AC.01` and `R049.§1.AC.02`

1. **`ClaimIndex`** registers all R049 claims under their canonical IDs and resolves R079's references against the local index. `R049.1.AC.01` resolves; `R049.1.AC.02` does NOT (no such claim defined) → linter emits `unresolved-reference`.

2. **Trace matrix** for R049 shows R079 in the Requirement column for `R049.1.AC.01` and `R049.3.LOCK.03`, `-` for `R049.3.LOCK.04` (no incoming references).

3. **VS Code hover** on `{R049.§3.LOCK.03}` in R079 displays the body of the LOCK.03 paragraph claim from R049 (lines 9–10 of R049.md). The refs panel in the hover shows the citing line from R079 with the FQID bolded.

4. **CLI `scepter trace R049`** renders the matrix; `scepter show R049.§3.LOCK.03` displays the claim body; `scepter gaps` flags `R049.1.AC.02` as a referenced-but-undefined claim.

Every contract in §1–§9 below specifies one cell of the cross-tab implicit in this example.

## Prior Art and Design Rationale

### Domain comparison

The claim reference grammar problem space has known solutions in adjacent domains. Each contributes a constraint to this spec.

| System | What it does | What this spec borrows | What this spec rejects |
|---|---|---|---|
| RFC 2119 keyword statements | Modal verbs (MUST/SHOULD/MAY) carry contractual force | Modal-status preservation in claim text; per-claim lifecycle | The unstructured-prose-only form — claims need addressable IDs |
| DOORS, ReqIF | Hierarchical requirement IDs with attributes and traceability | Hierarchical addressing (`NOTE.§N.PREFIX.NN`); attribute suffix (`:metadata`) | Heavyweight tooling, proprietary formats, manual IDs |
| Markdown wiki-link `[[Page]]` | Lightweight braceless reference | Braceless matching for note IDs (`{R004}` optional) | Page-only addressing — can't reach sub-document claims |
| JIRA ticket IDs (`PROJ-123`) | Compact, project-scoped IDs | Project-scoped uniqueness | Hyphen separator (collides with markdown ranges); flat namespace |
| Pandoc cross-references (`[@key]`) | Citation-key references | Inline references resolved at render time | Bibliography-only model — claims aren't citations |
| LSP / DAP | Stable wire-protocol grammar between tools | Per-shape behavior contract; multiple consumers conform to one grammar | The wire-protocol heaviness — markdown is the source of truth, not a serialization |

### Why this consolidating spec exists

Each upstream Requirement is internally coherent. {R004} defines the base grammar; {R005} extends with metadata; {R011} extends with cross-project; {R012} extends with ranges and adjacent-section binding. New extensions land as new Requirements that touch each affected consumer. This pattern has produced correct work but accumulates implicit behavior that no single document captures.

The accumulating risk is best illustrated by a real example. Suppose an author writes `**R049.LOCK.03**:` inside R049.md (a self-prefixed bold-paragraph-form definition) and a referenceable definition is intended. Today:
- The reference parser ({R004.§2.AC.04}, claim-parser.ts) accepts `R049.LOCK.03` as a valid `ClaimAddress`. So `{R049.LOCK.03}` from another note resolves grammatically.
- The definition parser (claim-tree.ts, derived from {R004.§3.AC.02}) does NOT accept `**R049.LOCK.03**:` as a definition because `LINE_CLAIM_RE` has no slot for a leading note-ID prefix.
- The claim therefore never enters the index. References to it fail. The linter emits `unresolved-reference`.

The asymmetry between reference and definition parsers is invisible from any single Requirement and visible only when the cross-tab is constructed. This spec constructs the cross-tab and exposes asymmetries like this one.

### Alternatives considered

| Alternative | Rejected because |
|---|---|
| Author one R-document per extension and never consolidate | Existing pattern. Produces invisible asymmetries between consumers as the surface grows. Maintenance cost grows with each extension. |
| Embed the cross-tab in {R004} | Inflates {R004} with content unrelated to base grammar. Every future extension requires re-opening R004. {R011}'s precedent (separate Requirement) argues against. |
| Maintain the cross-tab in claims.md (the agent skill) | claims.md is agent-facing, summarizing for use; not authoritative for behavior. The skill file would drift from canonical without a derived-from spec. {S001} establishes the precedent that agent-facing files derive from a Specification, not vice versa. |
| Define one Specification per consumer (parser-spec, hover-spec, etc.) | Would create N specifications that all redundantly enumerate every shape. The cross-tab structure is shape × consumer; either rows or columns must be the primary axis. Shape-primary (this spec's choice) places each behavior near its peers, simplifying impact analysis when a shape changes. |

### Methodological lineage

This spec follows the methodology in `claude/skills/scepter/artifacts/specification.md`. The cross-tab in §3 is closest to the Cleanroom "black box" — every consumer is observed at its boundary, with no claims about internal mechanism. Truth tables in §4–§7 follow the IEEE 830 / Volere "fit criterion" discipline: each cell is independently verifiable by inspecting the named consumer's output for the named input.

## Terminology

| Term | Definition |
|---|---|
| **Reference shape** | A syntactic form a reference can take. Distinguishes braced from braceless, qualified from bare, with-metadata from without, etc. NOT a single character pattern — a *family* of patterns sharing parsing rules. |
| **Definition shape** | A syntactic form a claim *definition* can take. Distinguishes heading-level from paragraph-level from table-row, bold-wrapped from plain, self-prefixed from bare. |
| **Permutation** | A specific combination of shape modifiers (e.g., "braceless self-prefixed paragraph-claim with metadata suffix"). The cross-tab dimensions multiply; not every permutation is meaningful. |
| **Consumer** | A subsystem that reads claim references or definitions and produces behavior. Includes parsers, the index, the linter, CLI commands, VS Code surfaces, the source-code scanner. NOT the markdown character stream itself. |
| **Self-prefix** | A note-ID prefix appearing as the leading segment of a *definition* inside that same note (e.g., `**R049.LOCK.03**:` inside R049.md). Distinct from a *reference* that uses the note-ID prefix to address a claim from elsewhere. |
| **Adjacent-section binding** | The parser-level rule (per {R012.§6}) that a bare section reference following a bare note reference within an allowed source-text gap is bound to the preceding note. |
| **Alias prefix** | A kebab-case prefix on a reference (`vendor-lib/R005.§1.AC.01`) declaring a cross-project citation. Per {R011}, a reference grammar element with read-only display semantics. |
| **FQID** | Fully qualified identifier — a claim address with note-ID, section path, and claim segments all present. The maximally unambiguous form. |
| **Canonical id** | The internal `id` string used by `ClaimNode` and `ClaimIndex`. Always section-qualified or bare; never carries note-ID prefix. The parser strips note-ID context before storing. |
| **Body** | The line range of content under a definition, computed by claim-tree.ts using the Rule 1/2/3 sequence in §2 below. |

## Data Model

The data shapes that flow through the pipeline. Field-level semantics (WHEN SET / WHEN NULL / INVARIANT) are documented for every nullable field; behaviors in §§1–9 reference these fields by name.

```typescript
/**
 * A parsed claim reference. Output of claim-parser.ts on a single
 * matched string. The reference grammar's atomic unit.
 *
 * @implements {R004.§1.AC.03} Fully qualified, partial, and bare claim resolution
 */
interface ClaimAddress {
  /**
   * WHEN SET: The reference begins with a recognized note ID
   *   (`[A-Z]{1,5}\d{3,5}`).
   * WHEN NULL: The reference is bare (no note prefix); resolution
   *   uses scope rules per the consuming context.
   * INVARIANT: When set, MUST match the note-ID grammar in {R004}.
   */
  noteId?: string;

  /**
   * WHEN SET: The reference contains numeric segments between note-ID
   *   and claim (e.g., `R004.§3.AC.01` → [3]; `R004.§3.1.AC.01` → [3,1]).
   * WHEN NULL: No section qualifier present.
   * INVARIANT: All elements MUST be ≥ 0 integers.
   */
  sectionPath?: number[];

  /**
   * WHEN SET: The reference targets a leaf claim (letter prefix + number).
   * WHEN NULL: The reference is a section-only or note-only address.
   * INVARIANT: When set, MUST match `[A-Z]+`. NEVER digits or mixed.
   */
  claimPrefix?: string;

  /**
   * WHEN SET: Numeric portion of the claim segment.
   * WHEN NULL: When `claimPrefix` is also null. INVARIANT: claimPrefix
   *   and claimNumber are jointly present or jointly absent.
   */
  claimNumber?: number;

  /**
   * WHEN SET: Sub-letter for refinement (`AC.03a`, `AC.03b`).
   * WHEN NULL: No refinement letter.
   * INVARIANT: Single lowercase letter when set.
   */
  claimSubLetter?: string;

  /**
   * WHEN SET: Colon-separated metadata items extracted from the suffix.
   * WHEN NULL: No metadata.
   * INVARIANT: Each item MUST satisfy the metadata charset
   *   `[A-Za-z0-9=_.§/-]+` per {R005} and {R011}'s alias-target relaxation.
   */
  metadata?: string[];

  /**
   * The literal source text that produced this address. Preserved for
   * downstream diagnostics that surface raw to the user.
   * INVARIANT: Always set; never null.
   */
  raw: string;

  /**
   * WHEN SET: Reference is cross-project; remainder resolves against
   *   the named peer per `projectAliases` config.
   * WHEN NULL: Reference is local-project.
   * INVARIANT: When set, MUST be a kebab-case alias name.
   * INVARIANT: When set, the address MUST NOT enter the local index,
   *   derivation graph, or trace matrix per {R011}.
   *
   * @implements {R011.§2.AC.01} alias-prefixed reference syntax
   */
  aliasPrefix?: string;
}

/**
 * A located reference — `ClaimAddress` plus where it appeared in source.
 * Output of `parseClaimReferences()` over a markdown content string.
 */
interface ClaimReference {
  address: ClaimAddress;
  /** 1-based line number in the source content. */
  line: number;
  /** 1-based column where the matched text begins. */
  column: number;
  /** True iff the reference appeared inside `{...}`. */
  braced: boolean;
}

/**
 * A node in the parsed claim tree. Output of claim-tree.ts.
 * Sections (interior nodes) and claims (leaves) share one type with a
 * `type` discriminator. Folder-note aggregation flattens trees from all
 * companion files into a single tree per parent note ID per {R008}.
 */
interface ClaimNode {
  type: 'section' | 'claim';
  /**
   * Canonical id within the containing note. Section path + claim
   * segment (e.g., `3.LOCK.03`); never carries the note-ID prefix.
   * INVARIANT: The note-ID is implicit — given by the containing note's
   *   filesystem identity. The index combines the two when registering.
   */
  id: string;
  sectionNumber?: number;
  claimPrefix?: string;
  claimNumber?: number;
  claimSubLetter?: string;
  /**
   * The full source-text span of the heading or paragraph line that
   * defined this node. Preserved verbatim; consumers display this
   * (with whatever inline formatting was present) in hover and CLI surfaces.
   */
  heading: string;
  /**
   * Markdown heading level (1–6) for heading-form definitions, or
   * `containingHeadingLevel + 1` for paragraph-form and table-row-form
   * definitions (pseudo-level).
   */
  headingLevel: number;
  /** 1-based line number of the heading or first line of the paragraph. */
  line: number;
  /**
   * Last 1-based line of the body. Computed by the Rule 1/2/3 sequence
   * documented in §2 of this spec.
   */
  endLine: number;
  children: ClaimNode[];
  /**
   * WHEN SET: Colon-separated metadata extracted from the definition.
   * Order is preserved.
   */
  metadata?: string[];
  /**
   * WHEN SET (PROPOSED, see §8): A note-ID prefix that appeared as the
   *   leading segment of the definition's source text.
   * WHEN NULL: Definition was bare (no note-ID prefix on the definition itself).
   * INVARIANT (PROPOSED): When set, the index MUST validate that this
   *   matches the containing note's ID; mismatch → `mismatched-self-prefix`
   *   error and the claim MUST NOT be registered.
   *
   * NOTE: This field is part of the §8 proposal and is not present in
   * the current implementation.
   */
  selfPrefix?: string;
}

/**
 * Errors emitted by the claim-tree builder and validator.
 * Each error type has a fixed semantics consumed by the linter and the
 * VS Code DiagnosticsProvider per {DD015.§1.DC.09}.
 *
 * @implements {R005.§2.AC.05} 'reference-to-removed' error type
 * @implements {R005.§2.AC.06} 'invalid-supersession-target' error type
 * @implements {R005.§2.AC.07} 'multiple-lifecycle' error type
 * @implements {R011.§2.AC.03} 'cross-project-derives' error type
 * @implements {R011.§2.AC.04} 'cross-project-superseded' error type
 */
interface ClaimTreeError {
  type:
    | 'duplicate'                       // Same id defined twice in same note (NOTE: tolerated as same-file repeat per R004 §1.AC.05 refinement; see §9)
    | 'non-monotonic'                   // Claim numbers out of order within a section
    | 'ambiguous'                       // Reserved (see §1 design decisions)
    | 'unresolved-reference'            // Reference does not resolve to any registered claim
    | 'forbidden-form'                  // AC01 (no dot), PH1.01 (alphanumeric prefix), FOO.AC.01 (multi-segment)
    | 'multiple-lifecycle'              // Two lifecycle tags on one claim
    | 'invalid-supersession-target'     // superseded=TARGET points at non-existent claim
    | 'reference-to-removed'            // Reference points at a `:removed` claim
    | 'unresolvable-derivation-target'  // derives=TARGET points at non-existent claim
    | 'cross-project-derives'           // derives=<alias>/<id> (forbidden per R011)
    | 'cross-project-superseded'        // superseded=<alias>/<id> (permanently forbidden per R011)
    | 'alias-unknown'                   // Alias name not in projectAliases config
    | 'peer-unresolved'                 // Alias resolves to peer project but ID not found
    | 'peer-target-not-found'           // Peer note exists but claim within it doesn't
    | 'mismatched-self-prefix';         // PROPOSED §8: self-prefix on definition doesn't match containing note ID
  claimId: string;
  line: number;
  message: string;
  conflictingLines?: number[];
  noteId?: string;
  noteFilePath?: string;
}
```

## §1 Reference Shape Catalog

Every shape a *reference* can take. References are read by `claim-parser.ts` (`parseClaimReferences`), produce `ClaimReference[]`, and are consumed downstream by the index for resolution and by every reading surface for display.

A reference is read in three steps:
1. Locate matching text (braced regex first, then braceless patterns).
2. Apply range-expansion check before address parsing.
3. Apply alias-prefix detection before component classification.

The shapes below are organized by braced-vs-braceless and by the qualification level. Each shape has fixed acceptance behavior and fixed downstream resolution.

### §1 narrative

References are how one note (or one source file) cites a claim defined elsewhere. The grammar grew from `{NOTE_ID}` (whole-note reference, original SCEpter feature) through claim-level addressing in {R004}, range expansion in {R012.§5}, and cross-project alias prefix in {R011}. Each extension maintains backward compatibility: every shape that worked before still works.

The catalog distinguishes three orthogonal axes:
- **Braced vs braceless** — whether the reference is wrapped in `{...}`.
- **Qualification depth** — note-only / section-only / claim-bare / claim-section-qualified / claim-fully-qualified.
- **Modifiers** — alias prefix, range suffix, metadata suffix, adjacent-section binding context.

A reference's permutation is a point in this product space. Not every permutation is meaningful (e.g., a range suffix on a note-only reference is undefined). The catalog enumerates only meaningful permutations.

### Reference shapes

§1.AC.01:5:derives=R004.§2.AC.02 The system MUST recognize braced references `{<address>}` for every address shape defined in §1.AC.03–§1.AC.10. Braced references MUST work regardless of any project configuration flag — they are always available. The `{` and `}` characters delimit the address; the inner text MUST be parsed as a single `ClaimAddress` or rejected as a unit.

§1.AC.02:4:derives=R004.§2.AC.05 The system MUST support a project configuration flag `claims.bracelessEnabled` (default: true) that controls whether unbraced references are recognized at all. When false, only braced references produce `ClaimReference` entries; bare text matching the braceless patterns MUST NOT be matched.

§1.AC.03:4:derives=R004.§1.AC.03 A note-only reference is a bare note ID (e.g., `R004`). It resolves to the entire note. Braceless matching MUST validate the leading shortcode against the project's configured note types (`knownShortcodes`); references to undefined shortcodes MUST NOT be recognized as note-only references.

§1.AC.04:4:derives=R004.§1.AC.03 A section-only reference is a `§N` form (e.g., `§3`, `§3.1`, `R004.§3`, `R004.3`). The `§` is optional emphasis ({R004.§2.AC.03}); presence MUST NOT change parsing behavior. A section-only reference resolves to the section node identified by the path; consumers MAY display the section heading and contents.

§1.AC.05:4:derives=R004.§1.AC.03 A bare claim reference is `PREFIX.NN` (e.g., `AC.01`, `LOCK.03`). It resolves within the current note's scope using the rules in {R004.§1.AC.04} (refined): ambiguity at definition time is NOT flagged; ambiguity is reported at resolution time when an actual reference fails to resolve uniquely.

§1.AC.06:4:derives=R004.§1.AC.03 A section-qualified claim reference is `§N.PREFIX.NN` (e.g., `§3.AC.01`). It resolves within the current note. The `§` MUST be optional. Numeric-only path forms (`3.AC.01`) MUST resolve identically.

§1.AC.07:4:derives=R004.§1.AC.03 A note-qualified claim reference is `NOTE.PREFIX.NN` (e.g., `R004.AC.01`). It resolves to the named note. When the named note has multiple sections defining `PREFIX.NN`, resolution MUST fail with the ambiguity surfaced as an `unresolved-reference` whose message names the conflicting fully-qualified candidates.

§1.AC.08:5:derives=R004.§1.AC.03 A fully qualified claim reference is `NOTE.§N.PREFIX.NN` (e.g., `R004.§1.AC.01`). It resolves uniquely. This form is the preferred shape in cross-document and cross-project contexts.

§1.AC.09:4:derives=R012.§5.AC.05 A range reference is `PREFIX.NN-MM` or `PREFIX.NN-PREFIX.MM` at any qualification level (`R004.§1.AC.01-06`, `{§1.AC.01-AC.06}`). The parser MUST expand to N individual `ClaimAddress` entries — one per integer in `[NN, MM]` inclusive — preserving the qualification of the leading address. Sub-letters are NOT supported in ranges. The expanded entries share `column`; downstream consumers MUST treat same-column siblings as expansion-related (per {R012.§6.AC.03}).

§1.AC.10:4:derives=R011.§2.AC.01 A cross-project alias-prefixed reference is `<alias>/<address>` (e.g., `vendor-lib/R005.§1.AC.01`). The alias MUST be kebab-case (`[a-z][a-z0-9-]*[a-z0-9]`). When an alias prefix is present, `aliasPrefix` is set on the resulting `ClaimAddress` and the remainder is parsed by the standard local rules. Transitive aliases (`a/b/R001`) MUST be rejected at parse time per {R011.§2.AC.07}.

§1.AC.11:3:derives=R005.§2.AC.04a A reference MAY carry a colon-suffix metadata block (`{R004.§1.AC.01:5:closed}`). The metadata MUST be extracted into `metadata: string[]` on the address. The first colon separates the claim path from metadata; subsequent colons separate items.

§1.AC.12:3:derives=R012.§6.AC.01 A bare section reference (`§5.2`) immediately following a bare note reference (`E032`) within the allowed source-text gap MUST be bound to the preceding note: `address.noteId` (and `aliasPrefix` if the previous reference carried one) is mutated to inherit from the predecessor; `address.raw` MUST stay as the literal source text the author wrote.

§1.AC.13:3:derives=R012.§6.AC.02 The allowed gap between a bare note reference and a bare section reference for adjacent-section binding is: optional `'s` possessive followed by required whitespace. Anything else (commas, parentheses, prose) MUST break adjacency and prevent binding.

§1.AC.14:3:derives=R012.§6.AC.03 When adjacent-section binding fires on a range expansion (`T057's §1.AC.01-02`), all expanded same-column siblings MUST inherit the binding when the first member binds.

## §2 Definition Shape Catalog

Every shape a claim *definition* can take. Definitions are read by `claim-tree.ts` (`buildClaimTree`), produce a `ClaimNode` tree, and are consumed downstream by the index for registration.

A definition is recognized in one of four positions: **heading**, **paragraph**, **table-row**, or (proposed) **self-prefixed paragraph**. Each position has its own regex gates, its own metadata-handling rules, and its own body-extent rules.

### §2 narrative

A definition introduces a claim into a note. Unlike a reference (which cites an already-defined claim), a definition is the *origin* of the claim's identity within the document. The parser must distinguish definition from reference reliably; if it confuses one for the other, the index registers nothing or registers the wrong thing, and every downstream surface fails silently.

The current grammar accepts four definition positions: heading-level, paragraph-level (with optional bold wrapping), table-row first-cell, and (proposed) self-prefixed paragraph. The four positions share a normalized internal representation (`ClaimNode`) but differ in their parsing rules and body-extent rules.

The cross-axis to position is *qualification* — whether the definition's source text carries a section or note prefix:

- **Bare**: `**LOCK.03**: ...` — claim segment only
- **Section-qualified**: `**§3.LOCK.03**: ...` — section + claim segments
- **Note-qualified self-prefix (PROPOSED §8)**: `**R049.LOCK.03**: ...` — note + claim segments

Claim-tree.ts currently accepts bare and section-qualified at all positions. It does not accept note-qualified self-prefix at any position. §8 proposes the self-prefix extension; this section catalogs current behavior plus the proposal location.

### Definition position 1: heading

§2.AC.01:5:derives=R004.§3.AC.02 A markdown heading whose text matches the claim regex MUST produce a claim `ClaimNode`. The regex MUST accept `(§N(.M)*\.)?(§)?[A-Z]+\.\d{2,3}([a-z])?` as the leading token. The `§` is optional emphasis on either the section path or the claim prefix and MUST NOT change semantics.

§2.AC.02:4:derives=R004.§3.AC.01 A markdown heading whose text matches the section regex `§\d+(\.\d+)*` MUST produce a section `ClaimNode`. Sections are interior nodes in the claim tree; claims are leaves nested under sections by heading-level hierarchy.

§2.AC.03:4 A heading-form claim's body extends from the heading line to the next markdown heading at same-or-shallower level (per claim-tree.ts Rule 2). Plain headings between two claims MUST also terminate the prior claim's body even when the plain heading itself produces no `ClaimNode`. Without this, intervening prose-headings leak content into the wrong claim's body.

### Definition position 2: paragraph

§2.AC.04:4:derives=R004.§3.AC.04 A non-heading line whose first non-whitespace token matches `(§N(.M)*\.)?(§)?[A-Z]+\.\d{2,3}([a-z])?` followed by `[\s:]` MUST produce a claim `ClaimNode` at heading level `containingHeadingLevel + 1` (pseudo-level). This is the paragraph-form definition.

§2.AC.05:4:derives=R004.§3.AC.04 A paragraph-form claim's body extends from the definition line to the next blank line, exclusive (per claim-tree.ts Rule 3). Without this rule, a paragraph-claim embedded inside a heading-claim's body silently absorbs unrelated content from the enclosing claim.

§2.AC.06:3:derives=R004.§3.AC.04 A paragraph-form definition MAY be wrapped in markdown bold (`**...**`) or italic (`*...*`, `_..._`, `__...__`) markers wrapping the claim ID portion. The parser MUST strip leading/trailing inline-formatting markers from the claim ID when matching. The unwrapped form (`AC.01 The parser MUST...`) and the wrapped form (`**AC.01** The parser MUST...`) MUST produce identical `ClaimNode`s.

### Definition position 3: table-row

§2.AC.07:3:derives=R004.§3.AC.04 When the project configuration enables table-row claim recognition (default: enabled), a markdown table row MUST be examined: the first cell's stripped contents (inline formatting removed) MUST be checked against the claim regex. When the first cell matches, the row produces a claim `ClaimNode` whose `heading` is the full row text (cells joined by ` | `).

§2.AC.08:3 An author MAY opt out of table-row claim recognition by including the directive `<!-- no-table-claims -->` anywhere in the document. When present, the parser MUST NOT recognize any table row as a claim definition.

### Definition position 4: self-prefixed paragraph (PROPOSED §8)

The fourth position is proposed (not currently implemented). See §8 below for the full proposal. It is included in this catalog for completeness so future readers see where the extension fits.

### Body extent rules

The claim-tree builder computes `endLine` for each `ClaimNode` using a three-rule sequence; the *minimum* of applicable caps wins.

§2.AC.09:4 Rule 1 (structural-sibling): the next structural node at same-or-shallower heading level terminates the body. If the next node is deeper, the parser MUST walk forward until it finds one at same-or-shallower level.

§2.AC.10:4 Rule 2 (heading-claim termination, applies only to non-paragraph nodes): the next markdown heading at same-or-shallower level terminates the body, even when that heading is plain prose with no claim/section pattern.

§2.AC.11:4 Rule 3 (paragraph-claim termination, applies only to paragraph-form and table-row-form nodes): the first blank line after the definition terminates the body, exclusive.

§2.AC.12:3 The defensive floor: `endLine` MUST NOT precede the claim's own `line`. If the rule sequence produces a smaller value, `endLine` MUST be set to `line`.

### Same-file repeat tolerance

§2.AC.13:3:derives=R004.§1.AC.04 When two definitions in the same note produce the same canonical id, the parser MUST keep the first occurrence (in document order) and silently drop subsequent occurrences. No `duplicate` error is emitted. This permits TOCs, summary tables, and appendices to restate claim IDs without producing lint noise.

## §3 Consumer Behavior Cross-Tab

The cross-tab itself. For every consumer in the system, this section specifies the contract that consumer MUST satisfy when it encounters each shape from §1 and §2.

The cross-tab is structured by *consumer*. Each subsection enumerates the shapes that consumer reads, the produces, and any consumer-specific behavior. The consumer-by-consumer organization makes impact analysis tractable: when a shape changes, the affected consumers' subsections are the audit list.

### §3.1 claim-tree.ts (definition parser)

Reads: markdown content. Produces: `ClaimNode` tree, `ClaimTreeError[]` (intrinsic errors only — non-monotonic, forbidden-form, alphanumeric-prefix, multi-segment-prefix).

§3.1.AC.01:4 The parser MUST be a pure function of (content) → (tree, errors). It MUST NOT take the note ID, the file path, or any cross-note context as input. Composability of the parser over arbitrary content fragments (snippets, hover previews, anonymous strings) is preserved by this rule.

§3.1.AC.02:3 The parser MUST emit `forbidden-form` errors for:
- Line-leading 2+-letter prefix without a dot (`AC01 The parser MUST...`)
- Alphanumeric prefix attempts (`PH1.01`, `PRD2.05`)
- Multi-letter-segment prefix attempts (`FOO.AC.01`)
- Each error MUST include the offending claim id, line number, and a remediation suggestion.

§3.1.AC.03:3:derives=R004.§1.AC.05 The parser MUST emit `non-monotonic` errors when claim numbers within a section are out of order, accounting for sub-letter ordering (`AC.03 < AC.03a < AC.03b < AC.04`).

§3.1.AC.04:3 The parser MUST NOT emit `unresolved-reference`, `cross-project-derives`, `cross-project-superseded`, `unresolvable-derivation-target`, or any other error that requires cross-note context. These errors are emitted by the index or linter, which have the necessary context.

### §3.2 claim-parser.ts (reference parser)

Reads: markdown content + `ClaimParseOptions` (knownShortcodes, bracelessEnabled). Produces: `ClaimReference[]`.

§3.2.AC.01:4:derives=R004.§2.AC.02 Braced references `{<address>}` MUST be matched first. Inside `{...}`, the entire string is parsed as a single address; failure to parse rejects the reference (no partial match).

§3.2.AC.02:4:derives=R004.§2.AC.05 Braceless matching MUST be skipped entirely when `bracelessEnabled` is false. When enabled, braceless patterns MUST be applied in order such that more-specific shapes are matched before less-specific shapes (alias-prefixed before plain note-qualified before bare).

§3.2.AC.03:3 References inside backticks (inline code spans) MUST be excluded. The parser MUST check whether the matched text is preceded or followed by a backtick at the immediate boundary.

§3.2.AC.04:3 References inside curly braces that are themselves nested inside another braced reference MUST be excluded from braceless matching. The parser MUST track brace depth at each scan position.

§3.2.AC.05:4:derives=R012.§6.AC.01 After all reference matching, the parser MUST run `bindAdjacentSectionRefs` on the produced reference list. Binding mutates `address.noteId` (and `aliasPrefix`) on bare section references that satisfy the source-text gap rule with a preceding bare note reference.

### §3.3 ClaimIndex

Reads: per-note `ClaimNode` trees, all `ClaimReference[]` from notes and source files. Produces: a queryable index keyed by canonical FQID; cross-references; trace-matrix data.

§3.3.AC.01:5 Each registered claim MUST be keyed by FQID (`<noteId>.<canonicalId>`). When the canonical id has section path, the FQID is `R004.1.AC.01`; when bare, the FQID is `R004.AC.01` (no section segment). Both forms MUST resolve to the same registry entry — the section-qualified form is canonical, the bare form is an alias.

§3.3.AC.02:4 The index MUST register only claims defined in the local project. Cross-project alias-prefixed references encountered in note prose or source code MUST NOT result in registered claims; they MUST be tracked in a separate cross-project citation list per {R011}.

§3.3.AC.03:4 For each `derives=TARGET` metadata on a registered claim, the index MUST validate that TARGET resolves to a registered local claim. Failure MUST produce an `unresolvable-derivation-target` error. Cross-project derives MUST produce `cross-project-derives` per {R011.§2.AC.03}.

§3.3.AC.04:4 For each `superseded=TARGET` metadata on a registered claim, the index MUST validate that TARGET resolves to a registered local claim. Failure MUST produce an `invalid-supersession-target` error. Cross-project superseded MUST produce `cross-project-superseded` per {R011.§2.AC.04} (permanent rule).

§3.3.AC.05:3 PROPOSED (§8): For each `ClaimNode` carrying `selfPrefix`, the index MUST compare it to the containing note's ID. Mismatch MUST produce `mismatched-self-prefix` and the claim MUST NOT be registered. Match MUST register normally.

§3.3.AC.06:3:derives=R008.§1.AC.01 For folder-based notes, the index MUST receive aggregated content from `NoteFileManager.getAggregatedContents()` (main file + companion files in alphabetical order, frontmatter stripped from companions). Claims defined in any sub-file MUST be registered under the parent note's ID. Section/claim id collisions across sub-files use the same-file-repeat rule (§2.AC.13).

### §3.4 Linter

Reads: `ClaimTreeError[]` from claim-tree.ts plus `ClaimIndex` cross-validation errors. Produces: human-readable lint output and machine-readable error stream.

§3.4.AC.01:4 The linter MUST surface every `ClaimTreeError` produced by the parser unmodified. The linter is a thin presentation layer over parser-emitted errors plus its own cross-validation set.

§3.4.AC.02:4 The linter MUST cross-validate references against the index. Every `ClaimReference` whose address fails to resolve MUST produce `unresolved-reference`. Resolution failure modes include: target note absent, target section absent, target claim absent, ambiguous bare resolution.

§3.4.AC.03:3 The linter MUST validate aliases against `projectAliases` config. Unknown aliases produce `alias-unknown`. Known aliases that fail to resolve produce `peer-unresolved`. Known aliases whose target peer note exists but the cited claim does not produce `peer-target-not-found`.

§3.4.AC.04:3:derives=R005.§2.AC.05 References to a `:removed` claim MUST produce `reference-to-removed`. Removed claims remain in the index for back-resolution but mark all incoming references as warnings.

§3.4.AC.05:3:derives=R005.§2.AC.07 A claim with two lifecycle tags (e.g., `:closed:deferred`) MUST produce `multiple-lifecycle`.

### §3.5 CLI commands

Reads: address arguments from user input. Writes: rendered output (table, JSON, tree).

§3.5.AC.01:4 `scepter show <address>` MUST accept any reference shape from §1. When the shape resolves to a section, output is the section heading and contents. When it resolves to a claim, output is the claim heading and body. When the shape includes glob patterns (`D*`, `T00[1-9]`), each expansion is shown.

§3.5.AC.02:4 `scepter trace <address>` MUST render a traceability matrix — one row per claim covered by the address, one column per projection (configurable note types). `-` indicates no coverage; populated cells list the citing notes.

§3.5.AC.03:4 `scepter trace <address>` MUST surface cross-project citations in a separate "Cross-project citations" footer, NOT as a projection column ({R011}).

§3.5.AC.04:3 `scepter gaps` MUST surface claims with partial coverage. Configurable filters: `--projection`, `--include-closed`, `--include-deferred`, `--importance N`. Removed claims MUST be excluded.

§3.5.AC.05:3 `scepter thread <address>` MUST produce a derivation tree following `derives=` chains both directions. Cross-project derivations are not followed (per {R011}).

§3.5.AC.06:3:derives=R007.§1.AC.01 `scepter search <pattern>` MUST search claim text using either substring or regex (`--regex`). Results MUST include the FQID, claim text, and source note.

### §3.6 VS Code surfaces

Reads: editor cursor position + open document content. Produces: hover content, decoration ranges, diagnostic markers, preview HTML.

§3.6.AC.01:4:derives=R012.§1.AC.01 The decoration provider MUST emit a badge adjacent to each claim *definition* in raw markdown view and markdown preview. The badge MUST encode source-coverage state and inbound reference count.

§3.6.AC.02:4:derives=R012.§1.AC.06 The badge MUST appear on definitions only. Citations of a claim MUST NOT receive a badge — only the canonical definition does.

§3.6.AC.03:4:derives=R012.§2.AC.01 The hover provider MUST distinguish three cursor positions: (a) on the claim's own definition heading/paragraph, (b) on a citation of a claim from elsewhere, (c) on a range expansion. Each produces a different hover layout.

§3.6.AC.04:4:derives=R012.§2.AC.04 The hover refs panel MUST distinguish source-code references from note-to-note references. Within Notes, references MUST be grouped by source note.

§3.6.AC.05:3:derives=R012.§3.AC.01 The markdown preview tooltip MUST appear on `mouseenter` of any `.scepter-ref` element. Stacked tooltips manage hide-timers per-level. Tooltip click navigation MUST trampoline through `.markdown-link` per {R012.§3.AC.05}.

§3.6.AC.06:3 The diagnostics provider MUST render every `ClaimTreeError` and linter error as a VS Code diagnostic at the indicated line. Severity follows the error type per {DD015.§1.DC.09}.

### §3.7 Source code scanner

Reads: source files from configured `sourceCodeIntegration.folders`. Produces: a source-reference index linking files to claim FQIDs.

§3.7.AC.01:4 The scanner MUST recognize `@implements`, `@validates`, `@depends-on`, `@addresses`, and `@see` followed by a brace-wrapped reference (`@implements {R004.§1.AC.01}`). Multi-claim compact form `@implements {R004.§1.AC.01,.AC.03,.AC.05}` MUST expand to N references all sharing the file location.

§3.7.AC.02:3 The scanner MUST recognize plain `{R004}` mentions in comments as `@see`-strength references (lower confidence than annotated references).

§3.7.AC.03:4 Cross-project alias-prefixed annotations (`@implements {vendor-lib/R005.§1.AC.01}`) MUST be recognized as cross-project citations and tracked separately from local-projection coverage. They MUST NOT count toward Source-projection coverage in the trace matrix per {R011}.

## §4 Metadata Suffix Behavior

Metadata is colon-suffixed onto a claim definition or reference. It carries importance, lifecycle, derivation, and freeform tags. The grammar evolved through {R004.§2.AC.04} (initial) → {R005.§2.AC.04a} (colon-separator refinement) → {R009.§4} (key-value with relaxed charset) → {R011.§2.AC.03–04} (alias-prefixed targets in derives/superseded).

§4.AC.01:4:derives=R005.§2.AC.04a Metadata items MUST be colon-separated (NOT comma-separated; the comma form was superseded). The first colon separates claim path from metadata; subsequent colons separate items.

§4.AC.02:4:derives=R005.§2.AC.04b Each metadata item MUST satisfy the charset `[A-Za-z0-9=_.§/-]+`. The `/` permits cross-project alias targets in `derives=` and `superseded=` values.

§4.AC.03:4:derives=R005.§2.AC.01 Lifecycle tags `closed`, `deferred`, `removed`, `superseded=TARGET` are mutually exclusive. At most one lifecycle tag MAY appear per claim. Multiple lifecycle tags MUST produce `multiple-lifecycle` per {R005.§2.AC.07}.

§4.AC.04:4 An importance tag is a single digit `1`–`5`. It MAY appear in any position in the metadata list (`:5:closed` and `:closed:5` are equivalent). When multiple importance tags appear, the parser MUST take the first.

§4.AC.05:3 A `derives=TARGET` item MUST have TARGET resolve to a registered local claim or alias-prefixed peer claim. Local resolution failure produces `unresolvable-derivation-target`; alias-prefixed targets produce `cross-project-derives` per {R011.§2.AC.03}.

§4.AC.06:3 A `superseded=TARGET` item MUST have TARGET resolve to a registered local claim. Alias-prefixed targets produce `cross-project-superseded` (permanently forbidden per {R011.§2.AC.04}).

§4.AC.07:3 Metadata items not matching any reserved key (importance digit, lifecycle tag, `derives=`, `superseded=`) are freeform tags, captured in `metadata: string[]` and surfaced in `show`/`trace` output but not validated for content.

§4.AC.08:3 Importance and lifecycle are independent metadata dimensions. A claim MAY carry both simultaneously (e.g., `:5:closed` or `:closed:5`). The mutual-exclusion constraint in §4.AC.03 applies *within* the lifecycle dimension only — a claim has at most one lifecycle tag, but its importance is orthogonal and may coexist with any lifecycle state. Freeform tags (§4.AC.07) and `derives=`/`superseded=` items are likewise independent dimensions.

## §5 Range Expansion Behavior

Per {R012.§5}. Range references expand at parse time into N individual addresses.

§5.AC.01:4 The compact range form `PREFIX.NN-MM` MUST expand to addresses `PREFIX.NN, PREFIX.NN+1, ..., PREFIX.MM` inclusive. NN MUST be less than MM; equal-or-decreasing ranges MUST be rejected.

§5.AC.02:4 The explicit range form `PREFIX.NN-PREFIX.MM` MUST behave identically to the compact form. Both prefixes MUST match.

§5.AC.03:3 Range expansion MUST preserve qualification. `R004.§1.AC.01-06` produces six addresses each carrying `noteId: "R004", sectionPath: [1]`. Alias prefix MUST also be preserved.

§5.AC.04:3 Sub-letters (`AC.03a`) MUST NOT appear in ranges. A range start or end with a sub-letter MUST be rejected.

§5.AC.05:3 Expanded addresses share the source `column`. Downstream consumers (adjacent-section binding, hover panel grouping) MUST treat same-column siblings as expansion-related.

§5.AC.06:3 Pad width for the numeric portion in the reconstructed `raw` of expanded addresses MUST match the original input padding. `AC.001-005` produces `AC.001, ..., AC.005`; `AC.01-05` produces `AC.01, ..., AC.05`.

## §6 Adjacent-Section Binding Behavior

Per {R012.§6}. Binds bare section references following bare note references.

§6.AC.01:3:derives=R012.§6.AC.01 Binding fires only when the previous distinct-column reference is a *bare* note reference (note-id only, no section, no claim) AND the current reference is a section-only reference (no noteId, has sectionPath). Mismatched preconditions break binding.

§6.AC.02:3:derives=R012.§6.AC.02 The allowed source-text gap is `(?:'s)?\s+` — optional possessive `'s` followed by required whitespace. Anything else (commas, parentheses, prose tokens) breaks binding.

§6.AC.03:3:derives=R012.§6.AC.03 When the section-only reference is itself a range expansion, all expanded same-column siblings inherit binding from the first member.

§6.AC.04:3:derives=R012.§6.AC.04 Only the first section reference after a note reference binds. A second section reference downstream of a non-allowed gap stops further binding chain.

§6.AC.05:3 Binding mutates `address.noteId` (and `address.aliasPrefix` if the previous reference carried one). `address.raw` MUST stay as the literal source text the author wrote — consumers that surface raw to users (diagnostics, hover) MUST echo what the author saw.

## §7 Cross-Project (Alias-Prefixed) Behavior

Per {R011}. Alias-prefixed references are read-only citations; peer claims do NOT enter the local index.

§7.AC.01:5:derives=R011.§2.AC.01 An alias prefix `<alias>/` at the start of a reference MUST be detected and stripped before standard address parsing. The remainder MUST itself parse as a valid local-shape address.

§7.AC.02:5:derives=R011.§2.AC.07 Transitive aliases (`a/b/R001`) MUST be rejected. The parser detects this by re-applying the alias regex to the remainder; a positive match MUST result in `null` from `parseClaimAddress`.

§7.AC.03:5 Alias-prefixed references MUST NOT enter the local claim index, derivation graph, gap report, or trace matrix as projection coverage. They are tracked in a separate cross-project citation list.

§7.AC.04:4:derives=R011.§2.AC.03 `derives=<alias>/<id>` MUST produce `cross-project-derives` (per-project derivation invariant; reconsideration permitted via future requirement).

§7.AC.05:5:derives=R011.§2.AC.04 `superseded=<alias>/<id>` MUST produce `cross-project-superseded` (permanently forbidden — local lacks lifecycle authority over peers).

§7.AC.06:3 The CLI `scepter show <alias>/<noteId>` MUST display the peer's note with a clearly visible peer-source header indicating which alias was resolved.

§7.AC.07:3 The trace matrix output MUST list cross-project citations in a separate "Cross-project citations" footer below the projection table. They MUST NOT appear as a projection column.

## §8 Self-Prefixed Definition Behavior

This section is the spec projection of {R004.§3.AC.05}. The behavioral contract below specifies the parser, index, and consumer behavior that R004.§3.AC.05 makes a binding requirement.

### Background

Authors occasionally want to write a claim definition that includes the note's own ID as a leading segment, typically wrapped in bold for emphasis:

```markdown
**R049.LOCK.03**: For migration sessions, the schema lock is held for the
duration of the session.
```

This form is grammatically distinguishable from a reference because it appears at line-leading position and carries body content. The reference parser already accepts `R049.LOCK.03` as a valid `ClaimAddress` (so external `{R049.LOCK.03}` resolves correctly *if* the definition is registered). The definition parser does not currently accept this form, producing an asymmetry: external citations parse but find no target.

### Architectural placement

The §8 proposal places the change in two layers:

- **Parser layer** (claim-tree.ts): recognize the self-prefix syntactically; record it as data on the node (`node.selfPrefix?: string`). Do NOT validate prefix-vs-containing-note here — the parser does not have file-system context.
- **Index layer** (claim-index.ts): on registration, compare `node.selfPrefix` to the containing note's ID. Match → register normally. Mismatch → emit `mismatched-self-prefix` and skip registration.

This placement preserves the §3.1.AC.01 invariant (parser is a pure function of content) while putting validation where the necessary context (note identity) already lives.

### Proposed acceptance criteria

§8.AC.01:4:derives=R004.§3.AC.05 The parser MUST accept an optional leading `[A-Z]{1,5}\d{3,5}\.` group on heading-form claim definitions and bold-wrapped paragraph-form claim definitions. Plain (non-bold) paragraph-form definitions MUST NOT accept the self-prefix to limit false-positive matching against in-prose references at line-leading position.

§8.AC.02:4:derives=R004.§3.AC.05 When the self-prefix is recognized, the parser MUST populate `node.selfPrefix` with the captured prefix string. The canonical `node.id` MUST be set to the bare or section-qualified form (no note-ID prefix), identical to what the bare form `**LOCK.03**` would produce.

§8.AC.03:4:derives=R004.§3.AC.05 The index MUST validate `node.selfPrefix` against the containing note ID at registration time. When prefix matches, the claim is registered normally. When prefix mismatches, the index MUST emit `mismatched-self-prefix` and MUST NOT register the claim.

§8.AC.04:3:derives=R004.§3.AC.05 The `mismatched-self-prefix` error MUST be added to `ClaimTreeError['type']` and surfaced through the same diagnostic pipeline as other parser/index errors. Linter, VS Code DiagnosticsProvider, and CLI lint output all consume it.

§8.AC.05:3:derives=R004.§3.AC.05 Section-qualified self-prefix definitions MUST be accepted: `**R049.§3.LOCK.03**:` produces `node.id = "3.LOCK.03"` and `node.selfPrefix = "R049"`. The behavior is identical to bare self-prefix except for the section path.

§8.AC.06:3:derives=R004.§3.AC.05 Self-prefixed definitions MUST work in folder-based notes. The "containing note ID" check uses the parent note's ID; companion sub-files defining `**R049.LOCK.03**:` inside the R049 folder match (per {R008}).

§8.AC.07:3:derives=R004.§3.AC.05 No change to reference parser behavior. References of the form `{R049.LOCK.03}` continue to parse as `{noteId: "R049", claimPrefix: "LOCK", claimNumber: 3}`. When the definition is registered (matched self-prefix or bare), the reference resolves; when the definition is rejected (mismatched self-prefix), the reference fails resolution and produces `unresolved-reference` per existing rules.

§8.AC.08:3:derives=R004.§3.AC.05 The `claims.md` skill file MUST be updated to document self-prefixed definitions in the "Authoring Claims" section, alongside the existing bare and section-qualified forms.

### Why bold-wrapping is required for paragraph-form

The paragraph-form parser must distinguish a definition (line begins with `R049.LOCK.03 ...`) from an in-prose reference (line begins with `R049.LOCK.03 is the canonical statement`). The bare and section-qualified forms have the same theoretical risk but in practice rarely appear at line-leading position outside of definition context. The note-prefixed form is much more likely to appear in cross-document discussion. Requiring bold-wrapping (`**R049.LOCK.03**: ...`) for the paragraph-form variant narrows the recognition surface to where authors actually want it, at the cost of one mandatory inline-formatting marker. Heading-form does not have this risk (headings unambiguously introduce content), so heading-form accepts plain self-prefix.

## §9 Folder-Note Aggregation Behavior

Per {R008}. Folder-based notes (`R001 Title/R001.md` plus companion `.md` files) are treated as a single logical document for claim purposes.

§9.AC.01:4:derives=R008.§1.AC.01 The claim index MUST receive aggregated content from `NoteFileManager.getAggregatedContents()`: the main file content followed by every companion `.md` file's content (frontmatter stripped from companions), companions ordered alphabetically by filename.

§9.AC.02:4:derives=R008.§1.AC.02 Claims defined in any companion sub-file MUST be registered under the parent note's ID. A `§2.AC.01` defined in `details.md` of the R001 folder is registered as `R001.§2.AC.01`.

§9.AC.03:4 Section and claim id collisions across sub-files MUST follow the same-file-repeat rule (§2.AC.13): first occurrence in aggregated order is canonical; subsequent occurrences are dropped silently.

§9.AC.04:3 Companion sub-files MUST NOT be independently referenceable. There is no `{R001/details.md}` syntax. References target the folder note as a whole or a specific claim within it.

§9.AC.05:3 Frontmatter on companion files MUST be stripped before aggregation. Only the main file's frontmatter is authoritative.

## Edge Cases

### Edge Case 1: Reference inside a paragraph-claim's body

**Trigger:** Author writes `### AC.01 Validation rule` followed by body prose `See {R005.§1.AC.03} for the lifecycle handling.`

**Detection:** The body line is scanned by the reference parser independently of claim-tree.ts. References within the claim's body MUST be captured as `ClaimReference` entries.

**Behavior:** The reference is registered as an outgoing edge from the containing note (NOT the containing claim — the claim is not the reference subject). The trace matrix shows R005's claim covered by the citing note, not by R005.AC.01 specifically.

### Edge Case 2: Range expansion crosses the alias prefix

**Trigger:** `{vendor-lib/R005.§1.AC.01-03}` — alias-prefixed range.

**Detection:** The range parser fires before the alias parser? No — the alias detection is part of `parseClaimAddress`. The range parser reads the `baseRef`, applies range expansion, then the resulting addresses share the alias prefix.

**Behavior:** Three expanded addresses, each with `aliasPrefix = "vendor-lib"`, `noteId = "R005"`, `sectionPath = [1]`, `claimPrefix = "AC"`, claim numbers 1, 2, 3 respectively. All three are tracked as cross-project citations; none enter the local index.

### Edge Case 3: Self-prefixed reference appears at line-leading position in prose (PROPOSED §8 context)

**Trigger:** Author writes `R049.LOCK.03 is the canonical lock authority claim.` as a paragraph line in any note.

**Detection:** The proposed §8.AC.01 requires bold-wrapping for paragraph-form self-prefix recognition. A plain self-prefix at line-leading position MUST NOT be recognized as a definition.

**Behavior:** The line is processed as prose. The reference parser still matches `R049.LOCK.03` as a `ClaimReference`. The claim-tree builder produces no `ClaimNode` from this line.

### Edge Case 4: Mismatched self-prefix in folder note

**Trigger:** Inside `R049/details.md` (companion file), author writes `**R051.LOCK.03**: text`.

**Detection:** Aggregation processes the companion as part of the R049 folder. The parser produces a node with `selfPrefix = "R051"`. The index compares to the parent note ID (R049). Mismatch.

**Behavior:** `mismatched-self-prefix` emitted. Claim NOT registered. References to `{R049.LOCK.03}` from elsewhere fail resolution; references to `{R051.LOCK.03}` also fail (R051 did not define it). The author must correct the prefix.

### Edge Case 5: Adjacent-section binding fires on a range across an alias

**Trigger:** `vendor-lib/R005's §1.AC.01-03` — possessive form, alias-prefixed note ref, section-qualified range follows.

**Detection:** The reference parser produces:
- Reference 1: `{aliasPrefix: "vendor-lib", noteId: "R005"}` (bare alias-note)
- References 2–4: `{sectionPath: [1], claimPrefix: "AC", claimNumber: 1..3}` (range expansion, same column)

The adjacent-section binding rule fires on Reference 2; Reference 2 inherits both `noteId = "R005"` and `aliasPrefix = "vendor-lib"`. References 3–4 inherit via same-column rule.

**Behavior:** Three cross-project citations to vendor-lib's R005.§1.AC.01..03. None enter the local index.

### Edge Case 6: Same-file repeat across sub-files in folder note

**Trigger:** Folder note R001 contains `01-core.md` and `02-extensions.md`. Both define `§2.AC.01`.

**Detection:** Aggregation produces concatenated content with `01-core.md` first (alphabetical). The first `§2.AC.01` definition is registered; the second is dropped silently per §9.AC.03 + §2.AC.13.

**Behavior:** Trace matrix shows one R001.§2.AC.01 — covered by whichever projections cite it. The author may not realize their second `§2.AC.01` was dropped. Mitigation: `scepter lint R001` reports section/claim counts; authors can verify expected counts match.

### Edge Case 7: Empty range

**Trigger:** `AC.01-01` (start equals end).

**Detection:** Range parser detects start ≥ end.

**Behavior:** Range is rejected (returns null from `tryExpandRange`). The text is not interpreted as a range. Falls through to standard single-address parsing — `AC.01-01` is not a valid address either, so the reference is silently dropped (no `ClaimReference` produced).

### Edge Case 8: Metadata key-value with `=` in target

**Trigger:** `:derives=R005.§1.AC.01` — target contains a dot, §, and digits.

**Detection:** Per §4.AC.02, the metadata charset `[A-Za-z0-9=_.§/-]+` permits all of these.

**Behavior:** Item is captured as the literal string `derives=R005.§1.AC.01`. The index parses the `=` to extract the key and target; the target then resolves through the normal address parser.

## Design Decisions

### Decision 1: Spec organized by shape primary, consumer secondary

**Decided:** §1 and §2 enumerate shapes; §3 enumerates consumers. Each consumer subsection then references shapes by ID.

**Alternatives considered:**
- Consumer-primary (one section per consumer, each enumerating all shapes): Was rejected because impact analysis when a shape changes needs to find every consumer affected, and shape-primary makes the rows of impact contiguous. The cost is some redundancy in §3 where shape-by-shape contracts re-cite §1/§2 ACs.
- Permutation-primary (one section per Cartesian product cell): Was rejected because the product space is too large and most cells are uninteresting.

**Rationale:** Future extensions add new shapes (§1 or §2) or new consumers (§3 subsections). Shape-additions touch §3 in many subsections; consumer-additions touch only the new subsection. The asymmetry favors shape-primary because shape-additions (range, adjacent-section, alias prefix, self-prefix) have been the recurring extension pattern. Consumer-additions (a new VS Code surface, a new CLI command) are less frequent.

### Decision 2: Self-prefix is bold-required for paragraph-form

**Decided:** Per §8.AC.01, self-prefixed paragraph-form definitions require bold-wrapping (`**R049.LOCK.03**: ...`). Plain (`R049.LOCK.03 ...`) at line-leading position is NOT recognized.

**Alternatives considered:**
- Accept plain self-prefix at line-leading (consistent with bare-form): Was rejected because cross-document references frequently appear at line-leading positions in prose ("R049.LOCK.03 is the canonical statement"). The bare-form risk is theoretical-but-rare; the self-prefix risk is theoretical-and-likely.
- Require colon (`R049.LOCK.03: text`): Was rejected because bare-form paragraph claims do NOT require a colon (`AC.02 The system MUST...`), and inconsistency between forms increases authoring confusion.
- Reject paragraph-form self-prefix entirely (only allow heading-form): Was rejected because the canonical use case (`**R049.LOCK.03**:`) is paragraph-form with strong existing usage in author intent.

**Rationale:** Bold-wrapping is a deliberate authorial gesture — the author has chosen to emphasize the FQID. Inverting that into a parser signal narrows the recognition surface to where authors actually want it. The cost is one mandatory inline-formatting marker for the proposed paragraph self-prefix form.

### Decision 3: Validation in index, not parser, for self-prefix matching

**Decided:** Per §8.AC.03, the parser records `selfPrefix` as data; the index validates against containing note ID and rejects mismatches.

**Alternatives considered:**
- Thread `noteId` into `buildClaimTree(content, noteId?)` and validate at parse time: Was rejected. It breaks parser purity (§3.1.AC.01), reduces composability for snippet/preview use cases, and puts a semantic check in a syntactic surface. The parser's name promises content→tree; the change would smuggle file-system identity into a pure function.
- Accept any self-prefix and silently treat as bare (forgive typos): Was rejected. This would let the parser silently rewrite authorial intent, which is a smuggling-shaped failure (claims.md §Smuggling). Better to reject hard with a clear diagnostic.
- Index as the prefix (R049 file's `**R051.LOCK.03**:` registers R051.LOCK.03): Was rejected because it violates the invariant that a claim is owned by its containing note. Allowing this would let one note unilaterally define claims of another.

**Rationale:** Each layer keeps its existing remit. Parser is pure; index applies semantic policy. The new error type joins the existing index-emitted error family.

### Decision 4: Folder-note aggregation alphabetical order

**Decided:** Per §9.AC.01 (deriving from {R008}), companion files are aggregated in alphabetical order by filename.

**Alternatives considered:**
- Manual order via frontmatter: Was rejected by {R008} because frontmatter is stripped from companions; only main file's frontmatter is authoritative.
- Order by file modification time: Was rejected because modification time is unstable and produces non-deterministic claim numbering across machines.
- Order by file size: Was rejected as arbitrary.

**Rationale:** Alphabetical order is deterministic and authorable — authors prefix files (`01-core.md`, `02-extensions.md`) to control logical order. The cost is a leading-digit convention; the benefit is no metadata overhead.

### Decision 5: Cross-project peer claims do NOT enter local index

**Decided:** Per §7.AC.03, alias-prefixed references are tracked separately from the local claim index.

**Alternatives considered:** See {R011} for the full design space. This decision restates the invariant from {R011}.

**Rationale:** Citation-not-federation invariant. The local project has no authority over peer claim lifecycle; the peer is unaware of incoming citations. Allowing peer claims into the local index would let local notes annotate peer claims with metadata the peer never opted into.
