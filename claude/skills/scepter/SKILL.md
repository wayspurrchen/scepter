# SCEpter

## NON-NEGOTIABLE RULES

These rules prevent the most common agent failures. Violating any of them will break the knowledge graph or produce wrong output.

1. **CONFIG FIRST** — Run `scepter config` before ANY action if you have not confirmed the note types and paths. Never assume note types, shortcodes, folder structures, or storage locations exist. In particular, never assume notes live under `_scepter/` — discovery paths are configurable and notes can live anywhere in the project.
2. **CLI ONLY** — Never use Read/Glob/Grep on note files for discovery. Always use `scepter ctx` commands (`list`, `show`, `search`, `gather`). The CLI knows where notes live; you don't. Exception: editing a file AFTER finding it with CLI (read the file first after using `scepter ctx create` to avoid update errors).
3. **NEVER GUESS IDs** — IDs are auto-generated by `scepter create`. Never predict the next ID, never manually create note files on disk, never pre-write references like `{D005}` before the note exists. Create first, read ID from output, then reference.
4. **GATHER BEFORE WORKING** — `scepter ctx gather [TASK_ID]` is MANDATORY before starting any task. Never work without full context.
5. **CREATE IMMEDIATELY** — Decision made → create decision note NOW. Question arises → create question note NOW. Don't defer.
6. **CONNECT EVERYTHING** — Every note must reference related notes. Isolated notes are useless. Build the graph actively.
7. **DATE EVERYTHING** — Run `date "+%Y-%m-%d"` before EVERY edit. Use the exact output. Never use "now", "today", or made-up dates.
8. **REFERENCE IN CODE** — All code implementing SCEpter notes MUST include `@implements`, `@depends-on`, `@see`, or `@addresses` comments with note IDs.
9. **USE CLAIM REFERENCES** — Use fully qualified claim paths: `{R012.§1.AC.15}` not bare `AC.15`. The dot between prefix and number is mandatory (`AC.01` not `AC01`). Hyphens are forbidden (`AC.01` not `AC-01`). **Read `claims.md` from this skill directory for the full claim system.**

## What is SCEpter?

SCEpter is a CLI-first knowledge management system that solves context loss and decision amnesia in AI-assisted development. It captures atomic units of project knowledge as markdown notes with unique IDs, creating a persistent knowledge graph through bidirectional references.

**Core problems solved:**
- Every new AI conversation starts with zero context
- Past decisions and rationale get buried
- AI recreates existing solutions
- Changes ripple without traceability

**How it works:** Notes → References → Knowledge Graph → CLI queries → Context for AI agents

**Key differentiator — Claim-level traceability:** SCEpter doesn't just link documents; it tracks individual claims (acceptance criteria, constraints, specifications) across projections. A requirement's `§2.AC.03` can be traced through the specification, into source code `@implements` annotations, and into test `@validates` markers. This is one of SCEpter's strongest value adds — agents are encouraged to use claim-level references (`{R004.§1.AC.03}`) over note-level references (`{R004}`) whenever the specific claim is known. See `claims.md` in this skill directory for the full system.

## Core Architecture

### Notes as Atomic Knowledge Units

Notes follow Zettelkasten principles — as small as possible to capture a core idea, as large as necessary to capture it fully. Each note has a unique ID and type, and lives in a type-specific folder.

**Note types** are configured per project in `scepter.config.json`:
```json
{
  "noteTypes": {
    "Decision": { "folder": "decisions", "shortcode": "D", "description": "..." },
    "Requirement": { "folder": "requirements", "shortcode": "R", "description": "..." }
  }
}
```

**Storage is configurable, not fixed.** Notes are discovered by their ID prefix across the `discoveryPaths` listed in `scepter.config.json`. In a default project they live under `_scepter/notes/[type_folder]/`, but a project may scatter notes across `docs/`, `specs/`, or the entire repo (`["."]`). **Always use `scepter config` and `scepter ctx list` to discover where notes actually are — never hard-code a path.**

The `_scepter/` folder itself still holds project-level state (config, templates, verification data), but it is not necessarily where notes live.

**Tasks** use shortcode `T` and sequential IDs (T001, T002...). The `Task` type (and its storage folder) should be defined in the project's `scepter.config.json` like any other type — check with `scepter config` first. Tasks bridge knowledge (decisions, requirements) and work (implementation).

### Configuration-Driven Everything

Every SCEpter project is unique. Never assume what types exist — always run `scepter config` first. Configuration defines note types, folder structure, source code scanning, and status validation.

**Status configuration** supports:
- **Status sets**: reusable groups of statuses (e.g., "workflow": ["pending", "in-progress", "completed"])
- **Per-type allowed statuses** with `suggest` (warn) or `enforce` (block) modes
- **Default values** applied when no status is provided

## Knowledge Graph

### Bidirectional References

When you write `{D001}` in a note, two things happen: your note gains an outgoing reference to D001, and D001 gains an incoming reference from your note. Both directions are queryable.

### Progressive Disclosure

Three levels of detail:
1. **List**: Quick overview (ID, title, status, reference counts)
2. **Show**: Full note content with reference excerpts
3. **Gather**: Origin note + all related notes following reference chains

Control depth with `--depth N`, limit with `--max-notes`.

## Source Code Integration

SCEpter tracks bidirectional references between notes and source code. In code, reference notes via comments:

```typescript
/**
 * @implements {R001} User authentication requirement
 * @depends-on {D005} JWT decision
 * @see {T025} Implementation task
 */
class AuthService { ... }
```

**Reference types:** `@implements`, `@depends-on`, `@addresses`, `@validates`, `@see`, plain `{ID}`.

**Claim-level references** use dot notation: `{R012.§1.AC.15}`. Compact syntax for multiples: `{R012.§1.AC.11,.AC.29,.AC.30}`. Range syntax is supported: `{R012.§1.AC.01-30}`. The dot between prefix and number is mandatory — `AC01` and `AC-01` are both invalid. See `claims.md` in this skill directory for the full syntax, rules, and examples.

**In tests:** `test('S002.§1.AC.01: eq matches identical values', () => { ... });`

**In documentation:** `This implements {R012.§1.AC.15} acceptance criteria.`

## The SCEpter Workflow Loop

1. **Capture** — Decision made → create note. Requirement found → create note. Question arises → create note.
2. **Connect** — Link decisions to requirements, tasks to specs, questions to answers.
3. **Use** — `scepter ctx gather` before starting work. Review all related context.
4. **Evolve** — Question answered → update with decision reference. Task done → mark status. Decision revised → new note referencing old.

## CLI Reference

Context subcommands work at the top level: `scepter list` = `scepter ctx list`.

### Discovery
```bash
scepter config                                    # ALWAYS run first - see types, shortcodes, modes
scepter list                                      # All notes (default: 25, newest first)
scepter list --limit 50                           # Override default limit
scepter list -t Decision,Requirement              # Filter by type
scepter list -t T --status pending                # Pending tasks
scepter list --modified-after "3 days ago"        # Recent activity
scepter list --created-after "1 week ago"         # Recently created
scepter list --tags auth,security                 # Filter by tags
scepter list --has-refs                           # Notes with references
scepter list --has-no-refs                        # Orphan notes
scepter list --references D001                    # What references D001?
scepter list --referenced-by T025                 # What does T025 reference?
scepter list --format tree --tree-depth 2         # Hierarchical view
scepter list --format json                        # Machine readable
scepter list --columns id,title,status            # Specific columns
```

### Viewing Notes
```bash
scepter show D001                                 # View a specific note
scepter show D001 -r                              # With references
scepter show D001 -r --depth 2                    # With reference chain (2 levels)
scepter show "D*"                                 # Glob: all decisions
scepter show "T00[1-9]"                           # Range: T001-T009
scepter show --source-file src/auth/service.ts    # What notes does this file reference?
```

### Context Gathering
```bash
scepter gather T001                               # EXTREMELY USEFUL; prefer over `show` for task bootstrapping
scepter gather T001 --depth 1                     # Direct references only
scepter gather T001 --depth 2                     # Two levels deep
scepter gather D001 --refs-only                   # Just reference structure
scepter gather T001 --max-notes 20                # Limit context size
```

### Searching
```bash
scepter search "authentication"                   # Full-text search in notes
scepter search "jwt" --include-source             # Search notes AND code
scepter search "auth|login" --regex               # Regex search
scepter search "api" --types Requirement           # Search specific types
scepter search "jwt" --context-lines 3            # Show context around matches
```

### Creating Notes
```bash
# ALWAYS search first to avoid duplicates
scepter search "caching strategy"
scepter list --types Decision --tags cache

# Create notes (ID is auto-generated - NEVER guess it)
scepter create Decision "Use Redis for caching" --tags cache,infrastructure
scepter create Requirement "Support OAuth2" --tags auth,security
scepter create Task "Implement login flow" --tags auth

# With initial content
scepter create Decision "Use JWT" --content "Based on {R001}, JWT provides stateless auth..."

# Folder-based notes (for complex notes with sub-documents)
scepter create Requirement "API Spec" --folder --tags api

# Convert between formats
scepter convert D001 --to-folder
scepter convert D001 --to-folder --dry-run
```

### After Creating a Note
The create command prints the file path and auto-generated ID. Use that path to edit content directly. Add `{ID}` references to connect to related notes.

### Lifecycle
```bash
scepter archive D001                              # Archive a note
scepter restore D001                              # Restore from archive
scepter delete D001                               # Soft delete
scepter purge D001                                # Permanent delete
```

### Help
```bash
scepter --help                                    # Main help
scepter ctx list --help                           # List command help
scepter ctx create --help                         # Create command help
```

## Status Management

### Frontmatter First Rule

All status changes MUST be in frontmatter `status` key BEFORE adding progress notes:

```yaml
---
status: in_progress  # ← UPDATE THIS FIRST
---

## Progress
- 2025-11-12: Changed to in_progress, started implementation
```

### Completion Rule

**NEVER mark tasks as `completed` or `done`** without explicit user verification.

Safe transitions: `pending` → `in_progress` → `blocked` → `in_progress` → `ready_for_review`

User-only transitions: → `completed`, → `done`, → `approved`

Check `scepter config` and `scepter types list` for project-specific allowed statuses.

## Common Anti-Patterns

### 1. Reading Note Files Directly

❌ `Read: _scepter/notes/decisions/D001.md` / `Glob: _scepter/**/*.md` / `Glob: docs/**/*.md`
✅ `scepter ctx show D001` / `scepter ctx list` / `scepter ctx search "term"`

**Why:** CLI tracks references, resolves types, provides context. Direct access breaks all of this. It also fails when the project uses non-default `discoveryPaths` — notes may live in `docs/`, `specs/`, or anywhere configured. Only the CLI knows the configured locations. If you genuinely need the path of a note you've already identified, `scepter show` prints it.

### 2. Creating Without Searching First

❌ `scepter create Decision "Use PostgreSQL"` (didn't check existing)
✅ `scepter search "postgresql"` then `scepter list --types Decision --tags database` then create only if no matches.

### 3. Isolated Notes (No References)

❌ `# D001 - Use React` / `We'll use React.` (no connections)
✅ `Based on {R001} for SPA requirement. Answers {Q003}. Implementation in {T030}.`

### 4. Fake Dates

❌ "Updated just now" / "Completed today" / Making up dates
✅ `date "+%Y-%m-%d"` → use exact output: `2025-11-12`

### 5. Self-Completing Tasks

❌ `status: completed` (set by AI)
✅ `status: ready_for_review` (let user verify and complete)

### 6. Guessing Note IDs

❌ Creating note then writing `{D005}` assuming that's the ID
✅ `scepter create Decision "..."` → read ID from output (might be D003!) → then use `{D003}` in references

### 7. Preserving Dead Provenance When Editing

❌ `### Query-expressible methods (currently classified as "distinct-purpose" — revised 2026-03-28)`
✅ `### Query-expressible methods`

**Why:** When correcting, updating, or rewriting a document, do NOT preserve the old wrong state inline. A note saying "this was previously classified as X" has zero value when the reclassification happened in the same session or has no downstream consumers who relied on the old classification. It wastes tokens, confuses future readers, and creates noise that looks like it might be load-bearing history when it isn't.

**The test:** Does the old state have a provenance chain that someone might need to trace? Did other documents, code, or decisions depend on the old classification? If yes, the history matters — capture it in a status update, a decision note, or a checkpoint. If no (the correction is within the same working session, or no one ever acted on the wrong version), just write the correct thing. The git history preserves what changed; the document should reflect what IS.

**This applies to:** Section headers, claim text, architectural descriptions, any prose being corrected. It does NOT apply to lifecycle-tagged claims (`:removed`, `:superseded`) — those use the claim lifecycle system, not inline annotations.

## Note Creation Guidelines

**Triggers:** Decision made → Decision note. Requirement discovered → Requirement note. Question arises → Question note. Work identified → Task.

**Timing:** IMMEDIATELY. Not "later."

**Quantity:** Create ONLY what was requested. Don't proliferate companion notes without permission.

**Content:** Capture what was STATED, not what you think should exist. Note questions to clarify rather than inventing specifications.

**After creation:** Edit the file to add content and `{ID}` references. Always date updates.
